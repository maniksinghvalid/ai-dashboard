## v1.0 Notes on Changed Requirements

### REQ-sentiment-engine — D8/D9 amendments (2026-05-13)

**Original acceptance:** sentiment refresh leg completes within a **5s budget** per refresh cycle, measured under `/benchmark`.

**Amended acceptance (effective at close):** sentiment leg uses `TIMEOUT_MS = 35_000`; cron exports `maxDuration = 60` (REQ-cron-three-tier-dag, D9). The 5s budget is **no longer** a release gate.

**Rationale:** D8 swapped the cardiffnlp roberta classifier for `meta-llama/Llama-3.3-70B-Instruct-Turbo` as an LLM-as-judge. Empirical accuracy uplift was significant; the cost was wall-time. The original 5s budget was a cardiffnlp-era assumption.

**Carry-over to v1.1:** the 35s timeout vs 60s function ceiling collision is a Phase-3 deferred concern (see v1.0-ROADMAP.md "Issues Deferred"). v1.1 should consider a tier-restructure or async sentiment leg if utilization climbs.

### REQ-cron-three-tier-dag — D9 amendment (2026-05-13) + Phase 3 hardening (2026-05-14)

**Original:** `maxDuration = 30`, ~12s expected wall-time.

**As shipped:**
- `maxDuration = 60` (D9).
- 15-45s expected wall-time depending on Tier 3 sentiment-leg traffic.
- Phase 3 added a distributed Redis lock (`cron:refresh:lock`, `SET NX EX 90`) wrapping `refreshAllFeeds()` so concurrent QStash retries return early with HTTP 200 `{status:"locked"}` instead of double-running.
- Phase 3 also made the Tier 1 source summary three-state (`written` / `skipped_empty` / `fetcher_threw`) instead of binary `ok` / `failed`. A run where every fetcher returns `[]` no longer false-greens.

### REQ-vitest-bootstrap — extended in Phase 4

**Phase 1 scope:** Node-environment unit tests for pure functions only.

**Phase 4 extension (additive):** added `jsdom`, `@testing-library/react`, `@testing-library/dom`, `@vitejs/plugin-react` devDeps; extended `vitest.config.ts` `include` glob to pick up `.tsx`; configured `plugins: [react()]` so JSX is transformed. 26 new component-render tests landed (full suite: 102 → 128).

## v2 / Deferred (carry-over to v1.1+ planning)

These items were explicit non-goals in v1.0 and remain open for v1.1 prioritization:

- **Spike-alert consumers** (toast, email, Slack) — `alerts:spikes` produces events only; no UI/notification surface.
- **Sentiment history view** (sparkline over hours/days) — v1.0 ships the live snapshot only.
- **Provider fallback chain** (e.g., Together → Replicate) — v1.0 is single-vendor per D7/D8. Rotation observability (Sentry-surfaced 401s) is the v1 stopgap.
- **`/api/sentiment` stale fallback** — v1.0 mirrors `/api/trending` (cold-miss 503); a stale-tolerant path is a separate decision.
- **Cron tier-restructure** for the sentiment 35s-vs-60s budget collision (Phase 3 deferred).
- **Stale upstream death monitoring** — per-source freshness probe + Sentry alerting + dashboard staleness UI (Phase 3 deferred).
- **Reddit OAuth tripwire** — calendar reminder set for 2026-05-20; if `[reddit] Non-200` rate >20% in prod, escalate to authenticated Reddit OAuth.
- **Real-Firefox visual verification** of the always-on thin native scrollbar (Phase 4 verifier's only residual `human_needed` item).
