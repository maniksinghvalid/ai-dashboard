<objective>
Replace the non-atomic daily-budget check-and-consume in `lib/api/sentiment.ts` with a single atomic operation so two concurrent runs cannot both pass the guard and together exceed `SENTIMENT_DAILY_CHAR_BUDGET` (P3 — High, SC-3).

Today `checkAndConsumeBudget` (`lib/api/sentiment.ts:76-85`) does `redis.get(key)` → `if (current + charsNeeded > budget) return false` → `redis.incrby(key, charsNeeded)` → `redis.expire(...)`. The gap between the `get` and the `incrby` is a double-spend window: two runs can both read the same `current`, both pass the check, both increment, and together overshoot the budget. Over the Upstash REST transport there is no MULTI/EXEC — atomicity must come from a single command or a single `eval`.

Purpose: the daily-char-budget circuit breaker (D1 / F1, a locked decision) actually holds under concurrency.
Output: `checkAndConsumeBudget` becomes a single atomic `redis.eval` Lua script (the recommended mechanism per 03-RESEARCH.md Open Question 3); the daily key format and 48h expiry are preserved; the 401 Sentry capture is untouched; co-located tests demonstrate the atomic guard rejects the second concurrent consumer.

This plan is a TDD plan: the test task runs FIRST (it pins the expected behavior of the atomic guard while the implementation is still the old non-atomic code — the new concurrent-consumer test is RED until Task 2 lands), then the implementation task makes it GREEN. This ordering keeps the riskiest change — the hand-written Lua `eval` rewrite — bracketed by a behavioral spec that already exists, instead of shipping the Lua first and testing it after.

This plan is fully independent of Plans 03-01, 03-02, 03-04 — it touches only `lib/api/sentiment.ts` and its test file. Wave 1.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cron Tier 3 → sentiment module | In-process call; `charsNeeded` is computed from already-fetched feed text, not raw attacker input. |
| sentiment module → Redis | The daily-budget counter key `sentiment:budget:<UTC-date>` and the new Lua `eval`. An attacker with direct Redis access could manipulate the counter. |
| sentiment module → Together AI | External API. Unchanged by P3 — the 401 Sentry capture is a regression guard. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-08 | Tampering / Injection | The Lua `eval` script | mitigate | The Lua script is a fixed module-level string constant — it is NEVER built by string-concatenating runtime values. The dynamic values (`charsNeeded`, `budget`, `BUDGET_EXPIRE_SECONDS`) are passed exclusively as `EVAL` `ARGV` parameters, not interpolated into the script body. This makes Lua injection structurally impossible. Enforced by acceptance criterion: the script is passed as a constant with `[charsNeeded, budget, BUDGET_EXPIRE_SECONDS]` as the args array. |
| T-3-09 | Tampering | The `sentiment:budget:<UTC-date>` Redis counter | accept | An attacker with direct Redis write access could zero the counter (defeating the budget cap) or inflate it (DoS-ing sentiment). This requires already-compromised Redis credentials — a separate, higher-tier concern; the budget counter is not the weak link once Redis itself is owned. The 48h key expiry bounds any malicious inflation to two days. |
| T-3-10 | Elevation of privilege (quota bypass) | concurrent double-spend of the char budget | mitigate | This is the exact bug P3 fixes. The single atomic `eval` (server-side check-then-incr in one Redis operation) closes the get/incrby race window so two concurrent runs cannot both pass. Verified by the concurrent-consumer test (eval resolves 1 then 0). |
| T-3-11 | Information disclosure / Spoofing / Repudiation / DoS | — | N/A | P3 introduces no new endpoint, no new external input surface, no new auth path. It is an internal atomicity refactor of an existing module-private function. The narrow `export` added for testability widens the module surface by exactly one already-internal function and introduces no new input boundary. |
</threat_model>
