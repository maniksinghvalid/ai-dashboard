<objective>
Add a distributed lock around `refreshAllFeeds()` so a second concurrent cron invocation — QStash retrying a failed/timed-out delivery — returns early instead of running the tiers and racing on ZSET windows + the sentiment budget (P2 — High, SC-2).

Purpose: QStash retries can no longer double-execute the 3-tier DAG. This also *moots* the ZSET member-collision precision bug (review Finding 6) — with no concurrent runs there is no sub-second double-count, so no separate ZSET fix is needed.
Output: a small `lib/cache/lock.ts` helper (`acquireLock` / `releaseLock`), a `cronLock` entry in `CACHE_KEYS`, the lock wrapping `refreshAllFeeds()` inside the function body so both `GET` and `POST` are covered, and co-located acquire/contend/release tests.

This plan depends on Plan 03-01 because both modify `app/api/cron/refresh/route.ts` — 03-01 rewrites the Tier 1 summary, 03-02 wraps the function body in the lock. Sequencing them avoids a merge conflict on the same file.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| QStash/Vercel-cron → cron route | Authenticated input (QStash signature / Bearer). P2 does not change auth; it adds a concurrency gate *after* auth. |
| cron route → Redis | The lock key `cron:refresh:lock` is a new Redis key used purely for in-cluster coordination. An attacker who already reached Redis directly could manipulate it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-04 | Tampering | The `cron:refresh:lock` Redis key | accept | An attacker with direct Redis write access could delete the lock (forcing a concurrent run) or set it (DoS-ing all refreshes). This requires already-compromised Redis credentials — at which point the cache itself is fully compromised and the lock is not the weak link. The 90s EX TTL bounds any malicious lock-set to 90s of refresh suppression. Disposition: accept; out of P2's blast radius (Redis-credential compromise is a separate, higher-tier concern). |
| T-3-05 | Information disclosure | The contended-path HTTP 200 `{ status: "locked" }` response body | mitigate | The contended response body is exactly the literal `{ status: "locked" }` — no lock value, no run-id, no timestamp, no internal state. The unique lock value (`Date.now()-uuid`) stays server-side in Redis and is never serialized into a response. Verified by acceptance criterion: the returned body is the fixed literal. |
| T-3-06 | Denial of service | QStash retry storm against the contended path | mitigate | Returning HTTP 200 (not 5xx) on contention is the deliberate mitigation — QStash treats 200 as success and does not retry, so a contended call cannot trigger a retry storm. A 5xx would re-contend forever. Enforced by acceptance criterion (`{ status: "locked" }` returned with status 200). |
| T-3-07 | Spoofing / Repudiation / Elevation | — | N/A | P2 adds no new endpoint, no new auth path, no privilege boundary. The lock runs strictly after the existing Bearer / QStash-signature checks. |
</threat_model>
