<objective>
Make the cron `summary` reflect whether Redis was actually written, not merely whether the fetcher resolved (P1 — Critical, SC-1).

Today `summary: "ok"` only means the Tier 1 fetcher promise fulfilled. An all-upstreams-dead run returns HTTP 200 with an all-green summary and writes nothing to Redis. The YouTube fetcher (`lib/api/youtube.ts`, the `if (videoIds.length === 0) return []` block) returns `[]` *before* `cacheSet` is ever called, so even the `[cache] skipping empty write` warn never fires for it.

Purpose: cron observability becomes honest — a dead system stops looking healthy. This is also the foundation a future monitoring phase builds on (alert on N consecutive `skipped_empty`).
Output: `cacheSet` returns a write boolean; a new pure `deriveSourceOutcome` helper maps each settled Tier 1 result to a three-state outcome; the cron `summary` uses that helper; co-located tests; updated CLAUDE.md Key Patterns notes.
</objective>
